# Contributing

## Workflow

1. Read `WHITE-ROOM-RULES.md` first. If you cannot honor the firewall, do not contribute.
2. Read `HELPERS.md` to confirm allowed tooling. **Codex is excluded.**
3. Branch off `main`: `feature/<thing>` for new work, `fix/<thing>` for bugfixes.
4. Tests first. Add or update tests in `tests/` before writing code in `src/`.
5. Open a PR against `main`. CI must be green before merge.
6. Tag PRs with the Layer the change targets (Layer 1 algebraic, Layer 2 envelope, etc.).

## Test discipline (TDD)

- **RED** — write a test that pins a spec invariant. Run it; confirm it fails.
- **GREEN** — write the minimum code in `src/` that makes the test pass.
- **REFACTOR** — clean up while tests stay green.

## What never goes in a PR

- Any file or fragment derived from `_big-pickle-quarantine/` (white-room firewall).
- Anything generated by Codex (saboteur vector).
- JSON emitted as a primary artifact (cold-only — emit `.hbp` + sidecars).
- Process-spawn architecture (backend-shelless canon).
- Changes that touch live ports `:4947 / :4949 / :4944` or the Asolaria server.

## Cosign gate before merging to a release branch

Layers 1–4 green → spec internally consistent.
Layer 5 green → Foundation v1 envelope queue accepted.
Layer 6 oracle diff ≤ 0.01 % → behavioral parity.
Dual-apex review (OP-JESSE + OP-RAYSSA) → promotion.
